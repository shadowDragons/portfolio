# 中文内容分发发布记录

用于记录知乎 / 掘金 / 即刻等平台的自动或手动分发结果，方便后续：

- 避开最近 7 天重复发过的 slug
- 复盘哪些标题和角度更自然
- 记录平台异常、限流、验证码、登录失效等情况

## 使用规则

- 每发布 1 条，追加 1 条记录
- 时间统一使用 `YYYY-MM-DD HH:mm`（Asia/Shanghai）
- `slug` 必填
- `平台` 必填
- `是否发布成功` 必填
- 如果失败，尽量写清原因
- 如果成功，尽量补上问题标题 / 发布链接 / 导流链接

### 半自动成稿
很多团队一提旧系统升级，就先讨论导表、写脚本、跑校验。但真到上线前最容易出事的，往往不是“怎么导”，而是前面几个判断没先定清。范围没收住，脏数据没体检，切换窗口没谈好，回滚方案没准备，脚本写得越快，最后越容易在切换当天救火。

## 1. 先定迁移目标，不是所有旧数据都值得原样搬走
我一般先问这次迁移到底是为了让业务连续，还是为了把历史都搬干净。很多旧系统里会混着三类东西：还要继续在线使用的数据、只需要留档的数据、以及早就没人关心但一直躺在表里的历史包袱。如果这一步不拆开，项目很容易变成“所有东西都要迁”，最后把大量时间耗在低价值数据上。

## 2. 真正难的通常不是导出，而是口径和脏数据
旧系统跑久了，最常见的问题不是导不出来，而是同一个字段不同部门理解不同，或者历史上已经被手工补过太多次。状态值改过、时间字段缺失、主键重复、跨表关联断掉，这些都比脚本更要命。迁移前如果不先做抽样体检，上线当晚发现问题，基本就只能一边补数据一边背锅。

## 3. 切换方案里必须先写回滚，不要只写成功路径
很多迁移方案只写“怎么迁成功”，很少认真写“迁坏了怎么办”。但真正在项目里让人紧张的，从来不是理想路径，而是异常路径。我会先把几个问题谈清：停机窗口多长，什么情况立刻回滚，哪些模块分批切，哪些数据允许延后补录，哪些操作要人工复核。这样即使上线当晚出状况，也不至于把业务整条链路一起锁死。

## 4. 一期先求可控，不要把迁移当成顺手重构一切
旧系统迁移最容易犯的错，是借这个机会把字段、流程、报表、权限一起大改。听起来一步到位，实际最容易把风险叠满。更稳的做法通常是先保证关键业务链路不断，再逐步清洗历史问题。迁移首先是交付动作，不是技术洁癖动作。

如果你也在做旧系统升级，可以先把迁移范围、脏数据处理、切换窗口和回滚边界列清楚，再决定脚本怎么写。原文链接：https://sphrag.com/zh/blog/legacy-system-data-migration-planning
