# 中文内容分发发布记录

用于记录知乎 / 掘金 / 即刻等平台的自动或手动分发结果，方便后续：

- 避开最近 7 天重复发过的 slug
- 复盘哪些标题和角度更自然
- 记录平台异常、限流、验证码、登录失效等情况

## 使用规则

- 每发布 1 条，追加 1 条记录
- 时间统一使用 `YYYY-MM-DD HH:mm`（Asia/Shanghai）
- `slug` 必填
- `平台` 必填
- `是否发布成功` 必填
- 如果失败，尽量写清原因
- 如果成功，尽量补上问题标题 / 发布链接 / 导流链接

- 摘要：很多团队一启动老 ERP 重构，就会争数据库先重做、接口先服务化、权限先重构。问题不是哪个听起来更底层，而是哪一层正在持续制造业务事故。我的经验是：多数情况下先收接口边界更稳；只有核心数据口径已经烂到接口层也兜不住时，数据库才该前置；权限不一定最先单独大改，但必须从第一阶段就作为硬约束进入方案。

### 半自动成稿
很多企业一决定重构老 ERP，第一反应就是先讨论顺序：数据库先重做，接口先服务化，还是权限先重构。

这三个方向听起来都对，所以项目一开始很容易开很多会。问题是，顺序一旦判断错，团队就会很快陷入另一种忙碌：底层改了不少，业务还是照样救火；规范做了很多，真正最失控的那一层却没先止血。

我自己的判断是，这不是一个“谁更底层谁先上”的问题，而是一个“哪一层已经在持续制造业务事故”的问题。

## 多数情况下，先收接口边界更稳

如果老 ERP 现在还在跑，最现实的约束通常不是技术，而是业务不能停。

这时候直接大动数据库，往往会把报表、同步脚本、外围系统、历史数据兼容一起卷进去。表面看是在治根，实际上很容易先把影响范围放大。尤其是老系统跑久了之后，很多调用关系根本不在文档里，而是在脚本、人工流程和各种历史补丁里。

所以多数情况下，我会更建议先收接口边界。先把几个最关键的问题定住：

- 哪些动作允许被外部系统调用
- 哪些字段可以写，哪些只能读
- 哪些状态流转必须经过明确流程
- 哪些高风险操作必须留痕

这样做的好处是，哪怕数据库暂时还没有彻底重整，系统至少先有了边界。很多原来被默认放行的脏写入、重复改状态、脚本直连数据库，都会在这一步先暴露出来。

对交付来说，这一步更像止血。它不一定最底层，但通常最适合先把混乱圈住。

## 数据库该先动的情况，通常是接口层已经兜不住了

当然也不是所有项目都该先动接口。

如果核心数据本身已经失真，比如同一个业务对象有多套主键、多套状态定义，跨表关系长期对不上，甚至同一份订单在不同模块里都不是同一个意思，那接口层再怎么包，也只是把脏地基包得更复杂。

这种情况下，数据库就该前置。但我还是不建议一上来就喊“全库重构”。

更实际的做法，通常是围绕一条主链路先收小范围地基。比如先把订单主表、明细、状态流转、库存占用这几个最容易传导错误的点理顺。先把一条链跑稳，比一次性推翻全部底层更接近可交付。

我见过不少项目的问题不是“不够现代”，而是业务事实已经对不齐。这时候数据库先动，不是为了架构好看，而是为了让系统重新有可信的数据主线。

## 权限不一定最先大改，但绝对不能最后才想起

老 ERP 里还有一层经常被低估，就是权限。

很多系统表面上看是有角色、有菜单、有按钮控制的，但真正关键的状态修改、字段写入、审批绕行，最后还是能被少数人或者少数脚本轻易改掉。结果就是前面刚把接口和数据理顺，后面又被越权操作重新打乱。

所以我的经验是，权限不一定要第一阶段就做成一套大而全的独立工程，但它必须从第一阶段就成为约束条件。

至少要先回答清楚几件事：

- 哪些角色能改关键状态
- 哪些字段必须审批后才能写
- 哪些动作必须留下明确审计记录
- 哪些默认管理员权限其实应该被收回

权限如果只留到后面“再统一做”，前面的重构成果通常很难稳住。

## 一个更实际的排顺序方法

如果一定要把这件事说得更落地一点，我一般会先看三件事：

第一，看现在最失控的是不是写操作入口。如果是，就先收接口边界。

### 半自动成稿
老 ERP 重构时，很多团队会先争一个问题：到底先统一主数据，还是先把报表口径拉齐？

这两个方向都没错。主数据不稳，后面所有流程都会被脏数据拖住；报表口径不统一，管理层看到的数字又会互相打架。但在真实交付里，我一般不建议一期同时把两件事都当成主战场，因为这很容易把项目做成一个永远收不完的治理工程。

## 这不是谁更底层的问题

主数据看起来更底层，所以很多人会下意识觉得应该先做主数据。这个判断有时候对，但不总是对。

我更习惯先问一个更具体的问题：现在最影响项目推进的，到底是“看不清”，还是“数据不真”？如果是管理层、财务、销售、生产对同一个指标各说各话，优先统一报表口径可能更现实；如果是物料、客户、供应商、仓库这些基础对象本身已经混乱到影响下单、库存和结算，那主数据就不能再往后放。

## 什么时候先拉齐报表口径

如果当前最大的痛点是大家都在看报表，但每个人看到的不是同一套数字，我会倾向于先拉齐口径。

比如订单额到底按下单时间算，还是按审核通过算；出货额按发货单算，还是按签收算；在制品、库存周转、应收这些指标分别取哪个状态。只要这些定义没说清，系统改得再勤，会议里还是会继续争“为什么你那边的数字和我这边不一样”。

这一步不一定要先把所有底层数据洗干净，而是先把几个关键指标的解释框架定下来。先让业务会议里大家说的是同一回事，后面的系统改造才有共同目标。

## 什么时候先动主数据

如果问题已经不是报表解释不一致，而是同一个物料有多套编码，同一个客户在系统里有多个名字，供应商、仓库、BOM、价格表之间经常对不上，那就应该把主数据提前。

这种情况下，先做报表很容易变成“把错误输入展示得更漂亮”。看板做出来了，数字也统一展示了，但订单、库存、采购和财务底层仍然串不起来，后面还是会靠人工 Excel 修正。

所以主数据先行的重点不是做一堆字段规范，而是先定三件事：唯一性规则、维护责任、变更流程。谁能新增，谁能修改，改完影响哪些业务链路，这些不定下来，ERP 重构很难真正变稳。

## 为什么不建议两头一起抓

报表口径统一，本质是在统一管理语言；主数据治理，本质是在重做数据责任边界。它们牵涉的人、规则和系统约束都不一样。

如果一期同时大推，常见结果是会议越来越多，规则越来越厚，但一线每天怎么录入、怎么审核、怎么改错反而没有稳定下来。看起来项目很完整，实际落地会变慢。

## 更稳的拆法

我一般会建议先选一个业务切口，而不是全公司一起铺开。

很多企业做流程管理时容易犯一个错：一开始就想把所有部门、所有表单、所有报表都纳进去。看起来很完整，但落地时会非常重。因为每个部门都觉得自己的需求重要，最后一期系统变成大杂烩，周期拉长，验收也很难说清楚到底算不算完成。

我做企业系统时，一般会先让团队选一条最值得优先跑通的链路。比如销售线索到合同，采购申请到入库，订单到发货，或者请假报销到财务处理。先把这条链路里的角色、状态、权限、异常情况拆清楚，再决定一期要做哪些页面和模块。这样比一开始按部门罗列功能更稳。

# 工厂 OA 推不动时，我一般先看流程例外和权限模型

做工厂 OA 或内部审批系统时，我不太会先判断“员工是不是不配合”。很多系统上线后没人爱用，表面看是培训没做好、执行不到位，实际往往是系统把现场工作想得太标准了。

办公室里的审批流程，很多时候可以按部门、岗位、审批层级往下拆。但工厂现场不一样，班组、车间、仓库、采购、质检、设备、行政之间，经常夹着临时插单、夜班交接、设备抢修、异常返工、代班处理。你如果只做一棵很规整的审批树，页面看起来很完整，现场用起来反而会很卡。

## 先分清标准流程和真实流程

OA 项目里一个常见误区，是把制度文件里的标准流程直接搬进系统。比如请假、报修、采购、质检异常，每条流程都画得很顺，但真实执行时一定会有例外。

更稳的做法，是先把流程分成三类：

- 高频且稳定的动作，可以固化成标准流程
- 高频但经常有例外的动作，要预留退回、转交、代办和备注
- 低频且变化很大的动作，不一定第一期就强行系统化

系统最该固化的不是每一句线下沟通，而是责任、状态和结果。否则大家最后会形成一个很尴尬的习惯：事情在线下聊完，系统只负责事后补记录。

## 权限不要只照抄组织架构

另一个容易踩坑的地方是权限。很多项目一开始会说，主管看本部门，经理看全部，行政能发起流程，听起来没问题，但真实业务经常不是按组织架构跑的。

设备维修可能按产线分工，质检异常可能按问题类型处理，采购补单可能需要仓库和计划同时可见，临时代班的人又可能只在某个班次拥有审批权。如果权限只按“部门 + 职级”做，很快就会出现该处理的人看不到、不该卡住的节点被卡住。

我更倾向于把权限拆成动作和场景：谁能发起，谁能查看，谁能编辑，谁能审批，谁能转交，谁能在异常情况下临时接管。权限模型拆细一点，后面推广阻力反而会小很多。

## 一期别把所有流程都搬上去

工厂 OA 最怕第一期就追求“大而全”：请假、报销、采购、维修、入库、出库、质检、用车、访客、宿舍、印章全部一起上。功能清单看起来很有成果，但每多一条流程，就多一套权限、通知、异常和培训成本。

如果基础流程模型还没跑顺，一次全上只会把混乱放大。我一般更建议先选 1 到 3 条最关键、最能体现协同价值、责任链又相对清楚的流程。先把发起、审批、催办、异常处理、统计和移动端使用习惯跑通，再扩第二批。

OA 真正能推下去，不是因为功能表很长，而是因为一线发现它没有额外添堵，管理层也能看到真实状态。能做到这一步，比第一版就做几十个流程更重要。

# 企业内部订单系统一期，先做 Web App 还是小程序？我一般看这 4 件事

做企业内部订单系统时，经常会遇到一个很早就被拿出来讨论的问题：一期到底先做 Web App，还是先做小程序？

这个问题表面上是在选入口，实际是在判断核心流程先落在哪一端。有人觉得内部系统就应该先有一个完整后台，也有人觉得员工和客户都在微信里，小程序更方便。但真到交付阶段，决定项目稳不稳的，通常不是哪个入口更流行，而是订单录入、审批、跟单、查询这些动作，到底发生在什么场景里。

如果一开始只按偏好选，很容易做成两边都别扭：小程序里塞满复杂表格，录入效率很低；Web App 里又堆了很多本该移动端完成的确认动作，最后用户还是回到微信群和 Excel。

## 先看订单录入是不是重操作

我一般会先看一期最核心的动作是不是“录单”。

如果订单字段很多，要填客户、商品、规格、数量、价格、折扣、交期、附件、备注，还要复制历史订单、批量修改、筛选状态、导出报表，那 Web App 通常更稳。

原因很简单：这类操作需要更大的屏幕、更密的信息展示、键盘输入、批量处理和多列表格。小程序不是不能做，但体验很容易被字段数量拖垮。最后页面一层套一层，用户每填一个订单都很费劲。

所以如果一期目标是让内勤、跟单、运营、财务把订单主流程跑起来，我更倾向于先做 Web App。先把数据结构、状态流转、权限和查询能力打稳，后面再补移动端轻操作，返工会少很多。

## 再看高频动作发生在电脑还是手机

反过来，如果订单的高频动作主要发生在移动场景，小程序就可能更适合作为一期入口。

比如销售在外面拜访客户时快速提交需求，门店现场录入简单订单，代理或客户只需要查进度、确认信息、上传照片、接收提醒，这类动作不一定需要完整后台。用户本来就在手机上完成沟通，让他回电脑开系统，反而会降低使用率。

但这里有个前提：小程序一期的动作要收窄。它更适合提交、确认、查询、提醒、拍照上传这类轻操作，不适合一上来承担完整订单管理、复杂配置和深层报表。

如果团队嘴上说“先做小程序，简单一点”，但需求清单里又有复杂录单、价格规则、权限管理、报表导出、后台配置，那其实不是小程序简单，而是把后台问题压进了小屏幕里。

## 权限和状态流转比入口更重要

订单系统真正麻烦的地方，经常不在页面，而在状态和权限。

一个订单从创建到完成，中间可能经过销售提交、内勤审核、财务确认、仓库备货、工厂排产、发货、售后处理。不同角色能看到什么、能改什么、能不能退回、能不能转交、异常时谁来处理，这些规则如果没拆清楚，先做哪一端都会乱。

Web App 的优势是更适合把后台管理、权限配置、列表筛选和异常处理放在一起。小程序的优势是更适合把某个节点的动作变轻，比如提醒某个人确认、让现场人员上传凭证、让客户查看进度。

所以我不太建议把问题问成“Web App 和小程序哪个更好”。更实际的问法是：一期最需要稳定的是整条订单链路，还是某几个移动端节点？前者更偏 Web App，后者更偏小程序。

## 更稳的做法：先选主战场，给另一端留接口

真实项目里，最怕一开始就说：“Web App 和小程序一起做，反正以后都要用。”

这句话听起来省事，实际很容易让一期范围失控。订单系统里每个角色都能提出自己那一端的需求，销售要移动录单，内勤要批量处理，老板要手机看报表，财务要导出，仓库要扫码，最后两端都做了，但核心链路没有一条真正跑顺。

我更建议一期先选主战场：

- 如果核心是内部录单、审批、跟单、查询和统计，先做 Web App
- 如果核心是外勤提交、客户确认、现场拍照和消息提醒，先做小程序
- 如果两端未来都要有，第一期也要先把主数据、接口、状态和权限模型设计清楚

这样做不是为了省功能，而是为了避免一开始就把系统拆散。只要底层数据和接口边界清楚，后面补另一端通常是扩展；如果一期只顾着把两个入口都做出来，后面修流程就可能变成重做。

## 结尾

企业内部订单系统一期先做 Web App 还是小程序，不该看哪个入口更顺眼，而应该看主流程在哪里、重操作在哪里、高频用户在哪里、后续维护边界在哪里。

我的经验是：先把最关键的一条订单链路跑稳，比同时做出两个入口更重要。入口可以补，数据结构、权限模型和状态流转一开始做歪了，后面修起来会很贵。

# 企业系统需求别先列功能，我一般先拆这 4 件事

做企业系统开发时，我现在不太建议一上来就开功能清单会。

很多项目刚开始会列出一长串模块：客户管理、订单管理、库存管理、审批流、报表、消息通知、权限后台。看起来需求很完整，但真进入开发后才发现，最难的不是少了哪个页面，而是流程、角色、状态和边界一开始没拆清。

功能列表只能说明“想要什么”，不能说明“业务到底怎么跑”。如果这个问题没先回答，功能越多，系统反而越容易做乱。

## 先拆核心流程，而不是先拆页面

我一般会先问一条最关键的业务链路：这件事从谁开始，到谁结束，中间经过哪些节点？

比如一个内部订单系统，不应该先讨论要不要订单列表、订单详情、导出按钮，而应该先确认：订单是谁创建的，谁审核，谁分配，谁执行，谁确认完成，异常由谁处理。

流程拆清以后，页面和按钮自然会长出来。反过来，如果只按页面拆，前期原型可能很好看，但后面经常出现一种情况：每个页面都有了，业务却接不起来。

## 把角色和权限单独拿出来看

企业系统最容易低估的，是角色和权限。

很多需求文档里会写一句“管理员可管理全部数据，普通用户只看自己的数据”，但真实项目里远没有这么简单。部门负责人能不能看下属数据？销售离职后客户归谁？审批人能不能修改申请内容？财务能看金额但不能改状态吗？

这些不是上线后再补的小细节，而是系统结构的一部分。权限如果一开始没设计清楚，后面每加一个角色，就可能牵动列表、详情、操作按钮、通知、日志和报表。

我一般会至少拆三类边界：谁能看数据，谁能改数据，谁对关键状态负责。只要这三件事含糊，系统后面就很难维护。

## 状态流转要比字段更早确认

很多系统需求会花大量时间整理字段：名称、类型、备注、附件、金额、时间。但真正让系统跑起来的，往往是状态。

一条记录从新建到完成，中间有哪些状态？哪些状态可以退回？哪些状态不能再改？谁能把它推进到下一步？异常状态要不要单独留痕？

这些问题如果不提前确认，后面就会出现很多临时判断：这个按钮先放着、这个状态先让管理员改、这个异常先手动处理。短期能绕过去，长期就会变成系统里最难清理的历史包袱。

对企业系统来说，状态不是展示文案，而是业务规则。状态流转拆得越清楚，代码、权限和操作日志都会更稳定。

## 一期先跑通核心链路，不要把系统一次做满

企业系统还有一个常见风险：第一期就想把所有模块都做完。

这通常不是技术上完全做不了，而是验证成本太高。流程还没跑过，权限还没被真实用户检验，报表口径也没稳定，就先把配置、统计、消息、导出、审批、移动端全部铺开，后面很容易一起返工。

我更倾向于先选一条最核心、最影响协作效率的链路，把它做到能录入、能流转、能追踪、能查状态、能留记录。只要这条链路跑稳，再补报表、配置项和次级模块，系统会稳很多。

一期不是越小越好，而是边界要清楚。核心链路必须完整，外围能力可以分阶段补。

## 结尾

企业系统开发前，真正该先梳理的不是“要多少个功能”，而是流程怎么走、角色怎么分、状态怎么变、一期先解决哪条核心链路。

功能清单当然要有，但它应该建立在这些判断之后。否则项目很容易变成页面很多、按钮很多，但业务一跑就到处补丁。

### 半自动成稿
# Shopify 模板商城扛不住 B2B 规则时，我一般先拆这 4 条边界

Shopify 这类模板商城很好用，尤其适合标准商品、公开价格、直接下单和快速上线。很多团队第一阶段用它把展示、购物车、支付和基础订单跑起来，是很合理的选择。

但我在项目里更怕另一种情况：早期明明是标准商城，后面逐渐加客户等级、合同价、询价转订单、销售审核、账期、区域库存、ERP 同步，最后所有规则都靠插件补。短期看好像省了开发成本，实际是把系统边界一点点补散了。

这时候要判断的不是“Shopify 好不好”，而是这套交易链路是不是已经从标准零售，变成了 B2B 订货系统。

## 1. 价格规则是不是已经进入主流程

如果所有客户看到的价格都差不多，优惠规则也比较简单，模板商城通常可以继续扛。问题出在价格开始按客户等级、区域、合同、币种、起订量甚至销售负责人变化。

这类规则如果分散在多个插件里，后面最难维护的不是展示价格，而是解释价格。某个客户为什么看到这个价格，历史订单当时用的是哪套规则，报价调整后旧订单怎么保留口径，这些都需要有稳定的数据归属。

我一般会把这里当成第一个分界点：如果价格只是前台展示的一部分，可以留在模板体系里；如果价格已经变成交易主数据，就应该考虑单独收口到 B2B 订货系统里。

## 2. 下单前是不是多了询价和审批

标准电商链路通常是看商品、加购、支付、生成订单。但 B2B 很多时候不是这样，客户可能先询价，销售再报价，内部再审核，财务确认账期，最后才转成订单。

这条链路一旦成为常态，就不只是多一个表单的问题。它会牵涉到状态流转、角色权限、修改记录、通知、超时处理和异常回退。

如果继续用几个插件拼，最容易出现的问题是：报价在一个地方，审批在另一个地方，订单状态又在第三个地方。出了问题之后，没人能快速判断当前到底卡在哪一步。

## 3. 订单后处理是不是依赖内部系统

很多 B2B 商城的真正难点不在前台页面，而在订单生成之后。订单要不要同步 ERP？库存以哪个系统为准？财务确认和仓库发货谁先谁后？销售跟进记录要不要回写 CRM？

如果订单只是平台内完成支付和发货，模板商城很合适。但如果订单后面要进入 ERP、仓库、财务、销售多个系统，就需要提前定义主系统和状态边界。

我更倾向于让订单主数据、客户价、审批、账期和内部处理状态归到同一个系统里。这样后续接 ERP、CRM 或报表时，至少知道哪边是源头，哪边只是展示或同步结果。

## 4. 继续加插件是不是已经比拆系统更贵

很多项目到临界点时，还会按“再加几个页面、再装几个插件”来估成本。这个估法很容易低估维护成本。

真正要看的不是插件多少钱，而是规则以后会不会频繁变化，角色会不会继续增加，异常订单怎么处理，历史记录能不能解释清楚，以及多个插件升级后会不会互相影响。

如果每次业务规则调整，都要在主题、插件、邮件通知、订单字段和外部系统之间来回补丁，那就说明问题已经不是页面开发，而是交易系统边界没有拆清楚。

## 我一般怎么拆

我不会一上来就建议推翻 Shopify。更稳的做法通常是先分三层：

第一层是展示和内容，比如品牌、产品介绍、案例、FAQ，这些可以继续放在官网或 Shopify 里。

第二层是标准交易，比如公开价格、直接下单、在线支付、普通订单处理，Shopify 仍然很适合。

第三层是 B2B 订货能力，比如客户专属价格、询价报价、审批、账期、订单状态、ERP 同步和内部处理，这部分如果已经是主流程，就应该单独设计系统边界。

这样拆不是为了把系统做复杂，而是为了避免所有规则都挤在模板和插件里。前台继续做它擅长的展示和标准交易，B2B 订货系统负责更稳定的规则、状态和协同。

如果一个 Shopify 商城已经开始靠大量插件承接客户价、审批、账期和内部订单流转，我会先停下来画一张完整交易链路图，再决定继续扩展模板，还是把 B2B 订货系统拆出来。原文里我也整理过这个判断思路，可以参考：https://sphrag.com/zh/blog/shopify-template-vs-custom-b2b-commerce

### 最终掘金正文
做企业系统报价时，我越来越不太建议只看“多少个页面、多少个功能”。

页面和功能当然要估，但真正容易在后面变贵的，往往不是页面本身，而是前期没说清的权限、数据、接口、异常处理和维护责任。

很多企业系统一开始看起来很简单：几个表单、几个列表、一个后台。等真正上线时才发现，问题不是“能不能做出来”，而是系统出了问题以后能不能查、数据错了能不能追、流程变了能不能改。

## 1. 权限不是菜单开关，而是责任边界

有些报价单里会把权限写成一个很轻的功能点，好像只是不同角色看到不同菜单。

但企业系统里的权限通常没这么简单。它至少会涉及几层：

- 页面能不能看
- 数据能不能看全
- 字段能不能编辑
- 状态能不能流转
- 操作之后要不要留痕
- 导出、删除、审批这类动作谁负责

如果这些边界报价阶段没拆清楚，后面最容易出现几种情况：有人能看到但不能改，有人能改但不该改，出了问题又查不到到底是谁改的。

我的习惯是，一期不一定要做很复杂的 RBAC，但至少要把最小权限模型说清楚。哪些角色必须存在，哪些关键操作必须留痕，哪些字段不能随便开放，这些都应该算进真实范围里。

## 2. 历史数据通常比新页面更麻烦

企业系统很少真的从零开始。

哪怕客户说“我们做一套新的”，背后也可能有 Excel、老 ERP、CRM、财务系统、人工台账，里面都有历史数据。

如果报价时只估新页面，不估数据迁移，后面很容易被这些事情拖住：

- 字段命名不一致
- 编码规则不统一
- 老数据有重复和缺失
- Excel 格式每个部门都不一样
- 导入后还要人工校验

这些工作看起来不像“功能开发”，但它们会实实在在占时间。尤其是系统要替换旧流程时，数据清洗、映射和校验最好单独评估，不要默认包含在“导入一下”里面。

## 3. 接口对接最怕只问能不能接

接口对接也很容易被低估。

很多沟通里只会问一句：“能不能和 ERP / CRM / 财务系统对接？”

从技术上讲，大多数时候都能接。但企业系统里更关键的问题不是能不能接，而是接完以后谁说了算。

比如：

- 哪个系统是主数据
- 哪边可以修改状态
- 接口失败后怎么重试
- 重复推送怎么处理
- 两边数据不一致时以谁为准
- 出错以后由谁排查

这些没写清楚，接口就会从一个开发任务变成长期维护坑。上线初期可能还能人工盯着，等业务量上来以后，每一次状态不同步都会变成扯皮。

所以我一般会把接口边界单独拿出来谈，而不是只把它写成“对接某某系统”。

## 4. 异常处理和运维交接决定系统能不能长期跑

演示环境里最容易被忽略的是异常。

正常流程当然都能跑通：提交、审批、通过、生成记录。但真实上线以后，经常遇到的是这些情况：

- 审批被退回后要不要重新提交
- 订单状态被多人同时修改
- 文件上传失败但表单已经保存
- 通知没送达
- 外部接口超时
- 用户误操作需要恢复

如果系统没有日志、没有基础告警、没有备份和恢复方式，最后很多问题都会变成“找开发改库”。

这类技术债在报价阶段很容易被忽略，因为它们不像页面那样容易展示。但从长期维护看，它们反而很关键。一个系统能不能稳定跑，不只看功能是否完整，也看出问题以后能不能定位、解释和恢复。

## 更稳的报价方式：把不确定性提前说出来

我觉得企业系统报价最怕两种情况。

一种是需求还没拆清楚，就为了好看给一个很低的总价。另一种是把所有未来可能变化都塞进一期，结果报价变重、周期变长，最后谁都不确定第一版到底要交付什么。

更现实的方式是把范围拆开：

- 哪些是一期必须跑通的闭环
- 哪些是已经确定的开发范围
- 哪些是有风险的数据和接口工作
- 哪些适合二期等流程稳定后再做
- 哪些维护责任必须提前约定

这不是故意拆项目，而是避免把所有未知都藏进一个报价里。

企业系统的长期成本，很多时候不是来自某个功能特别难，而是来自一开始边界太模糊。报价阶段把隐藏成本讲清楚，短期看可能不如低价方案好看，但后面返工、扯皮和维护压力会少很多。

# A B2B Ordering System Is Not Just a Storefront With Login

When a company first describes a B2B ordering system, the request often sounds simple: existing customers log in, browse products, choose specifications, and submit orders online.

That description is not wrong, but it is usually too shallow.

In real B2B projects, especially when the buyers are overseas distributors, large retailers, industrial customers, or long-term contract clients, the system is rarely just a private online shop. It becomes a mix of product data management, customer-specific pricing, permission control, quotation handling, order confirmation, and internal follow-up.

If phase one is designed like a normal ecommerce storefront, the first demo may look fine. Product list, detail page, cart, checkout, done. But once real buyers start using it, the missing boundaries appear quickly:

- one buyer can see files that should only belong to another buyer;
- a contract price is mixed with a reference price;
- a discontinued product can still be requested;
- a submitted order is treated as confirmed too early;
- sales still need to copy data back into chat, Excel, or ERP manually.

That is why I prefer to design a B2B ordering system from four boundaries before talking too much about pages: product data, pricing, permissions, and the post-submission order flow.

## 1. Product data is operational data, not only website content

In a public website, product content mostly explains the product. It helps visitors understand what the company sells and why it may be relevant.

In a B2B ordering system, product data has a heavier job.

The same product record may be used by buyers, sales, operations, warehouse staff, and follow-up teams. A product image or short description is only one part of it. The system may also need specifications, MOQ, packaging details, certification files, lead-time notes, replacement models, restricted countries, internal remarks, and lifecycle status.

This matters because not every field should behave the same way.

Some fields are for display. Some fields affect whether a buyer can submit a request. Some fields are internal and should never be shown to the customer. Some fields should trigger warnings for sales. If all of this is stored as one long product description, the system will become hard to maintain as soon as the ordering flow becomes real.

A better phase-one structure is usually to separate product data into at least three layers:

- display content that helps buyers understand the product;
- transaction data that affects inquiry, quotation, and order eligibility;
- internal handling data that supports sales and operations after submission.

This does not mean building a huge PIM system on day one. It simply means the product model should not be treated as a CMS page with a buy button attached.

## 2. Pricing needs a clear model before it becomes a support problem

Pricing is where many B2B ordering systems become messy.

In retail ecommerce, a product often has one visible price, maybe with promotions or variants. In B2B ordering, the same product may have different prices for different customers, regions, currencies, quantities, contracts, payment terms, and delivery conditions.

Some prices can be used directly. Some are only reference prices. Some require sales confirmation. Some depend on the buyer's contract. Some should expire after a specific quotation period.

If the system stores price as a single field on the product table, every exception becomes a patch later.

For large-buyer systems, I usually want the project team to answer a few questions early:

- Which prices can become confirmed orders without manual review?
- Which prices should only generate a quotation request?
- Do customer-specific prices override product-level prices?
- Are tiered prices based on quantity, customer level, region, or contract?
- How should historical quotations be kept for dispute handling?

The goal is not to make the first version complicated. The goal is to avoid pretending that all prices are the same kind of data.

A practical first phase can be simple: reference price, customer price, quotation-needed status, and manual sales adjustment. That is already much safer than forcing every case into one visible price field.

## 3. Permission design is about visibility and actions, not only login

Many B2B systems start with one assumption: after a customer logs in, they can see the private catalog.

That is often not enough.

A B2B customer is usually a company, not a single person. Procurement, engineering, finance, management, and regional branches may all need accounts. They may belong to the same buyer organization, but they should not necessarily see the same documents, prices, or actions.

For example:

- an engineering user may need technical documents but not pricing;
- a procurement user may see prices and submit inquiries;
- a regional branch may only see selected product lines;
- a finance contact may need invoices or order history but not new order submission;
- a distributor may see only the products they are authorized to sell.

If permission is only designed as logged-in versus not logged-in, the system will soon depend on one-off rules. Every large customer brings another exception, and the code becomes harder to reason about.

For phase one, I would rather keep the permission model small but explicit:

- buyer company;
- contact account;
- product visibility;
- price visibility;
- file access;
- inquiry or order submission permission;
- internal user responsibility after submission.

This does not require an enterprise-grade permission engine immediately. But the system should already know that seeing a product, seeing a price, downloading a file, submitting an inquiry, and confirming an order are different actions.

That distinction is what keeps the system maintainable when real customers arrive.

## 4. Submission is not the end of the order flow

A common demo mistake is to stop at “order submitted successfully.”

For internal delivery, that is only the beginning.

After a large buyer submits a request, several questions appear:

- Does sales need to confirm the price?
- Does stock need to be reserved?
- Is the lead time still valid?
- Are payment terms already approved?
- Can the buyer revise the order?
- Can sales reject or split the request?
- Does it need to sync with ERP?
- What happens if notification or sync fails?

In many B2B cases, submitted does not mean confirmed. It may mean “received,” “waiting for sales confirmation,” “quotation sent,” “buyer revision required,” “confirmed,” “cancelled,” or “moved to internal order processing.”

If these states are not designed, the team will recreate them outside the system through chat messages and spreadsheets. Then the ordering platform becomes a front-end form instead of a real workflow tool.

A safer first version should complete one traceable loop:

1. The buyer submits a request or order.
2. Sales reviews price, lead time, and terms.
3. The system records confirmation, rejection, or revision.
4. Internal users can track the current status.
5. The buyer receives clear feedback.
6. Failures in notification or integration have logs and an owner.

This loop may sound less impressive than a long feature list, but it is much more valuable. It proves that the system can carry real work, not just collect form data.

## 5. A better phase one is a boundary map, not a feature race

The first version of a B2B ordering system does not need to solve every future scenario.

It probably does not need a full ERP replacement. It may not need complex BI dashboards. It may not need deep automation across every warehouse, finance, and logistics step.

But it does need stable boundaries.

Before development starts, I would rather map these questions clearly:

- What product data is public, customer-only, transaction-related, or internal?
- Which pricing cases can be automated, and which need sales review?
- What can each buyer company and contact account see or do?
- What happens after a buyer submits an inquiry or order?
- Which steps belong inside phase one, and which should stay manual for now?
- Where do logs, notifications, and fallback ownership live?

This kind of planning may feel less exciting than designing a storefront. But in B2B systems, these boundaries are the product.

The real value is not that a buyer can click a button. The value is that the buyer, sales team, and internal operations can work from the same structured flow without constantly repairing missing rules manually.

# 旧系统数据迁移别急着写脚本，我一般先确认这 4 件事

很多旧系统升级项目，一聊到数据迁移，团队很容易马上进入技术细节：旧库怎么导，新库怎么接，脚本怎么跑，校验怎么做。

这些当然都要做，但我现在反而不太建议一开始就急着写迁移脚本。

因为真实项目里，数据迁移最容易翻车的地方，往往不是“导不出来”，而是前面几个判断没定清楚：到底迁哪些数据，旧字段在业务里到底是什么意思，历史脏数据怎么处理，上线切换失败后怎么退。

如果这些没先说清，脚本写得越快，可能只是更快把旧问题搬进新系统。

## 1. 先确认迁移范围：不是旧系统里有的都要搬

很多迁移项目一开始会默认一件事：旧系统里有什么，新系统就全部迁过去。

这个想法看起来稳妥，实际上很容易把范围拖大。

旧系统跑了很多年以后，里面会有大量历史痕迹：废弃字段、早期流程留下来的状态、已经不用的分类、只用于老报表的中间数据、甚至没人敢删但也没人再看的记录。

如果不先分层，迁移范围就会变成一个黑洞。最后团队花了很多时间，不是在迁核心业务数据，而是在想办法保留一堆新系统并不真正需要的历史包袱。

我一般会先把数据拆成几类：

- 新系统上线后还要继续编辑和流转的数据
- 只需要查询，不再继续修改的历史数据
- 只适合归档，不适合进入新系统主流程的数据
- 可以清理或合并的重复、无效、测试数据

这一步看起来不像开发工作，但它会直接决定后面脚本、表结构、校验规则和上线节奏。范围没收住，迁移就很难收住。

## 2. 再确认业务口径：字段名一样，不代表含义一样

旧系统迁移真正麻烦的地方，经常不是字段太多，而是字段含义已经变过很多次。

比如一个“状态”字段，早期可能只表示订单是否完成，后来又被拿来表示财务是否确认，再后来某些特殊订单又手工改过状态。表面上还是同一个字段，实际已经混了几套业务口径。

还有一些字段，看起来可以直接映射到新系统，但不同部门理解不一样。销售、仓库、财务看到同一个“确认时间”，心里想的可能不是同一个节点。

这类问题如果迁移前不拆清楚，上线后会很难解释：新系统明明按旧数据导入了，为什么报表不对？为什么流程状态不对？为什么某些历史订单看起来像还没处理？

所以我更习惯在迁移前先做口径表，而不是直接做字段映射表。

字段映射只回答“旧字段放到新字段哪里”；口径表还要回答：

- 这个字段在旧系统里真实代表什么
- 有没有历史阶段导致含义变化
- 新系统是否还需要保留原含义
- 遇到不一致时以哪个业务规则为准

这一步如果偷懒，后面很多“数据 bug”其实都不是技术 bug，而是口径没对齐。

## 3. 脏数据要提前暴露，不要等上线当天才发现

旧系统迁移前，我一定会建议先做数据体检。

不是简单看总行数，也不是只看能不能导出，而是抽样看异常值、空值、重复值、断开的关联关系、状态不合法、时间字段缺失、联系人信息不完整这类问题。

老系统里的脏数据很正常。问题不在于有没有脏数据，而在于项目组有没有提前知道它们在哪里，以及准备怎么处理。

常见的处理方式大概有几种：

- 能自动修正的，写清楚清洗规则
- 不能自动判断的，交给业务人工确认
- 不影响新系统主流程的，作为历史备注或归档保留
- 会影响上线的，高优先级提前处理，不要拖到切换当天

我比较怕的是迁移方案里只写“导入成功多少条”，但没写“异常数据怎么处理”。因为上线当天真正卡住人的，往往就是那一小批异常数据。

## 4. 回滚和人工兜底要跟脚本一起设计

很多迁移方案只写成功路径：几点停旧系统，几点跑脚本，几点校验，几点切新系统。

但旧系统迁移最需要提前设计的，恰恰是失败路径。

比如校验发现关键数据缺失怎么办？部分模块迁过去了，但业务发现口径不对怎么办？新系统上线后半天内出现无法继续处理的异常，能不能退回旧系统？哪些数据允许补录，哪些数据必须冻结？

这些问题不要等出事后再开会。

我通常会希望迁移方案里至少提前写清楚：

- 什么情况下必须回滚
- 回滚后新系统里已产生的数据怎么处理
- 哪些模块可以分批切换，哪些不能拆开
- 哪些数据需要人工复核后再进入主流程
- 上线后第一天谁负责盯校验结果和异常处理

回滚方案不是为了真的回滚，而是为了让团队知道边界在哪里。没有回滚边界的迁移，本质上就是把所有压力都压到上线当天。

## 最后

数据迁移很容易被低估，因为它看起来像一件“把旧数据搬到新系统”的技术活。

但从交付角度看，它其实更像一次边界确认：哪些历史要保留，哪些问题要清洗，哪些口径要重定，哪些风险要允许人工兜底。

所以旧系统升级时，我一般不会先问脚本多久能写完，而是先问这 4 件事有没有定清楚：范围、口径、脏数据、回滚。

这些定清楚了，迁移脚本才是在执行方案；这些没定清楚，脚本再快，也只是把不确定性搬到新系统里。

# 内部系统重构时，前端值不值得重做？我通常先看这 4 个判断

很多内部系统重构，一开始最容易把注意力都放在后端：数据库要不要拆，接口要不要重写，服务要不要拆分，性能瓶颈在哪里。

这些当然重要，但如果项目已经进入多人协作、复杂表单、角色权限和状态流转阶段，真正每天折磨用户和维护团队的，往往不是“接口能不能通”，而是前端到底有没有把业务规则接住。

我现在判断一个内部系统重构时，前端值不值得重做，通常不会先看框架名词，而是先看下面这 4 个问题。

## 1. 先看复杂度，不要只看页面数量

很多团队评估前端工作量时，习惯先数页面：列表页几个，详情页几个，编辑页几个。

但内部系统真正贵的地方，经常不是页面数，而是表单和状态数。一个采购单、报价单、工单或者审批单，背后可能就有草稿、提交、退回、补资料、再次审批、作废这些不同状态。每种状态下，哪些字段可见，哪些字段可改，哪些按钮能点，规则都不一样。

如果这些判断长期散落在各个页面里，短期也许能跑，但后面只要改一个字段、补一个异常提示、调整一个角色权限，就很容易牵一发而动全身。

所以我更建议单独评估三件事：

- 表单复杂度有多高
- 状态流转有多少种
- 同一个业务对象的可见、可改、可提交规则有没有统一整理过

这三件事比“总共有几页”更能决定前端是不是已经到了该重做的阶段。

## 2. 再看前端有没有工程边界，而不是纠结用哪个框架

很多人讨论重构前端，第一反应是问：要不要上 React，要不要换 Vue，要不要补一套组件库。

我不觉得这些问题最关键。框架只是工具，真正决定后续维护成本的，是前端有没有工程边界。

比如页面状态、请求状态、权限状态是不是混在一起；表格、筛选、弹窗、导入导出有没有一致做法；错误提示是只把接口报错吐出来，还是能让业务用户看懂为什么不能继续操作。

如果这些边界没有建立起来，就算换了新框架，最后也可能只是把旧系统从“老页面”换成“新页面”。界面看起来更新了，但维护时还是一样怕改，一样靠经验猜哪里会受影响。

现代前端对内部系统真正的价值，不是把页面做得更潮，而是把复杂界面变成可维护的工程。

## 3. 不一定一次推翻，先挑一个高频流程做闭环

很多老系统都不适合一次性全量替换。范围太大、依赖太多、切换风险也太高。

更稳的做法通常是先挑一个高频、痛点明确、边界相对清楚的流程，先把前端闭环跑通。这个闭环至少要包含列表、详情、编辑、审批、异常提示和操作反馈，而不是只做一个“新版页面壳”。

这样做的好处是，真实问题会更早暴露出来：

- 老接口到底够不够用
- 状态字段是不是已经混乱
- 权限规则有没有说清
- 用户是否接受新的交互方式

等这一个流程跑顺以后，再反推组件规范、权限处理和接口约定，会比一开始就写一套很大的重构规范更实际。

## 4. 也要承认，不是每个后台都值得完整重做前端

不是所有内部系统都值得投入一轮完整前端重构。

如果这个系统本来就使用频率不高、流程很简单、只有少数管理员偶尔维护数据，那很多时候修几个明显的可用性问题、补日志、补权限校验，可能已经够了。

但如果系统每天都被多人协作使用，表单复杂、角色多、异常频繁、培训成本高，那前端就不是“皮肤层”问题了。它会直接决定业务规则能不能稳定执行，也决定后续维护是不是每次都要重新摸逻辑。

这类系统如果前端一直停留在“接口调好了再拼页面”的思路里，最后通常会出现一种很典型的结果：后端越来越现代，前端还是越来越难改。

## 最后

我现在看内部系统前端重构，核心不是追新技术，而是判断它有没有进入这样一个阶段：业务规则已经复杂到，前端如果不工程化，后面每次改动都会变成风险。

所以比起先问“要不要换框架”，我更习惯先问这 4 个问题：表单和状态复杂不复杂，前端边界清不清，高频流程能不能先跑一个闭环，这个系统到底值不值得完整重做。

这几个判断先做清楚，前端重构才是在解决交付问题，而不是单纯换一层新界面。

问题通常不是模型不够强，而是第一步选错了入口。企业做 AI，先做哪个，不是在选功能，而是在选改造成本、验证难度和组织阻力。

## 1. 资料散、问答重复时，先做知识问答更稳

如果当前最大的痛点是文档散、制度版本乱、新人总要到处问人，那知识问答通常最适合作为第一站。它的好处是不用一开始就改主业务流程，先把资料检索、权限范围和回答边界跑顺，就能很快看出有没有价值。

如果企业已经有比较清楚的流程节点、输入字段和处理规则，比如询盘分发、工单归类、资料初审、报销校验，这类环节做 AI 自动化，通常比做一个“万能助手”更容易看到结果。

销售助手、采购助手、客服助手看起来最像“真正把 AI 用起来了”，所以也最容易让管理层感兴趣。但这类项目一期最容易失控，因为它同时要求上下文完整、角色目标一致、建议可解释，还可能碰到系统写回和责任归属。

# Workflow Automation Is Not Ready for Launch Until Failure Handling Is Designed

In automation projects, teams usually spend most of their energy proving that the workflow can run by itself.

That sounds reasonable, but it often points attention at the wrong milestone.

A workflow is not production-ready just because it can complete the happy path without human help. It becomes production-ready when the team knows what happens after a timeout, a duplicate trigger, a partial write, a rule conflict, or a manual state correction in the middle of the process.

That distinction matters because many automation projects look convincing in demos and still create more operational stress after launch. The system appears to be saving work, but people are still watching it, checking edge cases, fixing wrong states, and explaining unexpected outcomes to other teams.

Over time, I have become much less interested in whether a workflow can be automated in theory. I care more about whether the failure path is understandable, owned, and recoverable. If that part is still vague, the workflow is not really automated. It is manual work with a cleaner screenshot.

## 1. A working demo hides the most expensive part of automation

Most demos are built around the clean path. The data is complete. The record is in the right state. The downstream interface responds normally. No one edits anything halfway through. No external service is delayed. Under those conditions, almost any workflow can look smoother than it will in production.

Real operations are different. A field is missing. Someone submits twice. A status changes between trigger time and write time. A third-party API accepts one part of the request and drops another. A human operator overrides a value that the automation was still relying on. None of these cases are unusual. They are exactly what makes operations operational.

This is why I do not treat "the API is connected" as a serious readiness signal. The better question is whether the team can clearly describe the main failure classes before launch. If the answer is still fuzzy, pushing for more automatic coverage usually increases risk instead of reducing labor.

## 2. Human fallback is not a notification. It is a process node

One of the weakest patterns I see is a workflow that claims to have fallback, but the real plan is just to send an alert when something fails.

An alert is not a fallback strategy.

If a workflow fails, somebody needs to know who takes over, how quickly they must act, what they are allowed to change, whether they should retry the automation, and whether the workflow can safely resume afterward. Without those rules, the automation does not hand work back to humans cleanly. It just pushes confusion into another inbox.

I prefer to treat fallback as part of the workflow design itself. There should be a default owner for exception cases, an escalation path when the owner does not respond, a small set of explicit operator actions, and a clear difference between two situations: manual confirmation that lets the automation continue, and manual takeover that stops the automatic path completely.

That may sound operationally heavy for a first release. In practice, it usually makes the first release much lighter to run. Teams stop improvising every time something goes wrong, and that alone removes a surprising amount of hidden cost.

## 3. Audit and rollback are part of business trust, not only engineering hygiene

As soon as automation starts changing status, generating records, sending notifications, or syncing to another system, traceability stops being a technical nice-to-have.

The business needs to answer basic questions afterward. What triggered this action? Which inputs were used? What did the system decide? What exactly changed? Did a human adjust it later? Which version of the rule or prompt was active at the time?

If those questions cannot be answered quickly, every incident turns into guesswork. The technical team digs through logs. Operations reconstruct events from chat messages. Stakeholders debate what the system "probably" did. Trust erodes long before the automation is turned off.

Rollback belongs in the same conversation. Not every action can be reversed in the same way, and pretending otherwise is another common mistake. Some writes are safely reversible. Some need compensation steps. Some should never be auto-reversed but still need before-change snapshots and a clear manual repair path.

That design work should happen before launch, not after the first visible mistake. If rollback only exists as a vague promise that "engineering can fix it in the database," then the workflow is carrying more risk than the demo suggests.

## 4. Phase one should prove one recoverable loop, not a full automated chain

The most reliable automation projects I have seen did not try to automate everything at once.

They picked one workflow segment with three properties: high enough frequency to matter, stable enough rules to validate, and failure cost low enough that the team could learn safely. Then they delivered the whole loop around that segment: trigger, execution, exception path, operator takeover, logs, and recovery rules.

That kind of phase one is less flashy than an end-to-end automation pitch, but it produces much better evidence. The team learns which rules are truly stable, which exceptions happen often enough to deserve product treatment, and which steps should keep human confirmation even if the rest of the workflow becomes more automatic later.

This is also where the economics become clearer. Automation saves money only when it reduces repeat work over time. If it simply replaces predictable manual effort with unpredictable manual firefighting, the organization has not gained efficiency. It has only changed the shape of the labor.

The most practical first question, then, is not "How much of this workflow can we automate?" It is "Can we automate one bounded part of it in a way the team can understand, trust, and recover from?"

That is usually the difference between a workflow that survives production and one that stays trapped in the demo phase.
